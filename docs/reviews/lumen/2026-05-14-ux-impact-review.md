# SEW UX 임팩트 리뷰 - 2026-05-14

작성자: 루멘(Codex)
검증 방식: `localhost:3000` dev 서버 실행 후 Codex in-app Browser Playwright로 직접 체험
관점: 기능 통과가 아니라 놀라움, 마찰, 신뢰, 재방문 이유

## 요약

SEW는 이미 단순 퀴즈 앱을 넘어섰다. 가장 강한 차별점은 기출, 개념, 진단, 오답, 기록이 서로 연결되는 구조다. 특히 기출 문제에서 관련 개념으로 이동하고, 개념 페이지에서 다시 기출 이력을 보는 흐름은 특수교육 임용생에게 "이 앱은 내 시험을 안다"는 감각을 줄 수 있다.

하지만 현재 가장 큰 약점은 내부 데이터 언어가 사용자 화면에 새어 나오는 점이다. `special-education-act`, `pbs`, `laws`, `visual-impairment`, `5000%` 같은 표현은 서비스 신뢰를 직접 깎는다. 경험의 골격은 좋고, 감탄 포인트도 있으나 마지막 polish가 사용자 몰입을 방해한다.

## 여정 1. 첫 방문자 온보딩에서 오늘 미션까지

경로: `/` 신규 온보딩 -> 시험일 -> 학습 수준 -> 약점 과목 -> 하루 목표 -> 계획 생성 -> 홈 미션

### Wow Point
- D-day, 학습 수준, 취약 과목, 하루 목표를 묻는 순서가 좋다. 임용 준비생에게 바로 개인화된다는 느낌을 준다.
- 계획 생성 후 홈이 `오늘의 미션 0/3`으로 단순화되어 있어 첫 행동이 명확하다.
- "퀴즈 풀기, 개념 학습, 용어 학습"의 3단 구성은 하루 학습 루틴으로 설득력이 있다.

### Friction
- 온보딩에서 `행동지원`, `관련 법령`을 취약 과목으로 선택했지만 홈은 `오늘 집중: 특수교육학`으로 표시됐다.
- 사용자는 "내가 선택한 값이 반영됐나?"라고 의심할 수 있다.

### Trust Risk
- 개인화 입력 직후 반영 결과가 다르게 보이는 것은 신뢰 리스크다.

### Quick Win
- 계획 생성 직후 홈 상단에 "선택한 취약 과목: 행동지원, 관련 법령" 또는 "오늘은 행동지원부터 시작합니다"를 보여준다.
- 자동 집중 과목이 별도 로직이면 이유를 표시한다. 예: "첫날은 기본 진단을 위해 특수교육학부터 시작합니다."

## 여정 2. 진단평가 후 약점과 추천 확인

경로: `/diagnosis` -> `/quiz/ox` -> 10문항 풀이 -> 결과 화면 -> 홈 복귀

### Wow Point
- 결과 화면이 단순 점수에서 끝나지 않고 챕터별 분석, 유형별 분석, 취약 영역 TOP 5, 추천 학습까지 이어진다.
- "틀린 문제 다시 풀기"가 있어 행동 전환이 자연스럽다.
- 진단 후 약점 개념으로 바로 이동할 수 있는 구조가 좋다.

### Friction
- 결과 화면과 챕터 분석에 `special-education-act`, `pbs`, `visual-training`, `theory` 같은 내부 slug가 그대로 노출된다.
- 임용생 입장에서는 전문성보다 미완성 데이터베이스를 보는 느낌을 받을 수 있다.

### Trust Risk
- 사용자가 결과를 신뢰하려면 분석 라벨이 사용자 언어여야 한다. 내부 slug는 분석 품질 전체를 의심하게 만든다.

### Quick Win
- 챕터/개념 slug를 한글 display title로 변환하는 공통 formatter를 적용한다.
- 결과 화면의 추천 학습은 "관련 법령 > 특수교육법 총칙"처럼 과목과 개념명을 함께 보여준다.

## 여정 3. 오늘학습 OX 루틴

경로: `/daily` -> STEP 1 OX 퀴즈 -> 10문항 답변 -> 채점 -> 단답형 전환

### Wow Point
- 채점 후 각 문제별 해설 품질이 높다. 틀린 이유와 핵심 개념이 즉시 연결된다.
- "틀린 영역 OX 다시 풀기"와 "단답형으로 넘어가기"가 있어 복습과 다음 단계가 명확하다.

### Friction
- STEP 1에서 10문항이 한 화면에 길게 펼쳐진다. 학습 밀도는 높지만 모바일 사용자는 압도감을 느낄 수 있다.
- 문제 위 링크가 `ebd`, `aba`, `disability-types`, `intervention`처럼 표시된다.
- 채점 후 화면이 매우 길어져 "내가 지금 다음에 뭘 하면 되는지"가 아래쪽에 묻힌다.

### Missed Delight
- "8/10, intervention과 general-curriculum 보강 필요"가 나오지만, 이를 오늘 미션 완료감이나 개인화된 한 문장으로 감싸지 않는다.

### Quick Win
- Daily OX는 한 문제씩 또는 3문제 단위로 접어 보여주는 모드를 검토한다.
- 채점 직후 상단 sticky summary를 둔다. 예: "8/10 완료. 타임아웃과 교수적 수정 위계만 다시 보면 됩니다."
- 내부 slug 링크를 한글 개념명으로 변환한다.

## 여정 4. 기출에서 개념으로 이동

경로: `/kice` -> 2027 예상 문제 -> 관련 개념 학습 -> `/concepts/관련 법령/특수교육법총칙과국가의무`

### Wow Point
- 기출 문제 아래에 "관련 개념 학습"이 직접 붙어 있어 매우 강력하다.
- 개념 페이지에는 기출키워드, 기출 포인트, 이 개념의 기출문제 18건이 함께 나온다.
- 이 흐름은 SEW의 핵심 차별점이다. 임용생이 "기출과 개념이 따로 놀지 않는다"는 감각을 받을 수 있다.

### Friction
- 기출 페이지의 첫 화면 정보량이 많아 사용자가 "문제 풀기, 분석 보기, 개념 보기" 중 어떤 행동을 해야 할지 순간적으로 갈릴 수 있다.
- 개념 페이지 상단에는 좋은 설명이 있지만 "이 개념으로 바로 문제 풀기" CTA가 첫 화면에는 약하다.

### Quick Win
- 개념 페이지 상단에 "이 개념 기출 18건 풀기", "관련 OX 5문제", "오늘 미션에 추가" 같은 바로 행동 CTA를 둔다.
- KICE 관련 개념 박스는 아주 좋으므로, 이 디자인 패턴을 퀴즈 결과/오답노트에도 통일한다.

## 여정 5. 기록과 성장감 확인

경로: 학습 수행 후 `/record`

### Wow Point
- D-day, 오늘 문제 수, 연속 학습, XP, 주간 학습, 보강 과목, 오답, 플래시카드가 한 곳에 모인다.
- "지금 하면 좋은 것"이 있어 재방문 행동을 만들 수 있다.
- 배지와 레벨 구조는 장기 사용 동기와 잘 맞는다.

### Friction
- `introduction`, `behavior-support`, `physical-disability`, `laws`, `communication-disorder` 등 내부 과목명이 그대로 노출된다.
- 최근 틀린 문제도 `visual-impairment visual-training`, `laws theory`처럼 표시된다.

### Trust Risk
- 플래시카드 전체 정답률이 `5000%`로 표시됐다. 이건 즉시 고쳐야 하는 P0 UX 신뢰 이슈다.
- 기록 화면은 사용자가 성장감을 확인하는 곳이므로 숫자 오류가 있으면 전체 앱의 신뢰가 크게 흔들린다.

### Quick Win
- 정답률 계산의 denominator를 확인하고 0~100% clamp 또는 올바른 누적 계산으로 수정한다.
- 모든 subject/chapter slug를 display title로 변환하는 공통 매핑을 기록 화면에 적용한다.

## 우선순위

### P0. 신뢰 손상 즉시 수정
- `/record` 플래시카드 전체 정답률 `5000%`.
- 기록/진단/Daily 화면의 내부 slug 노출.
- 온보딩 취약 과목과 홈 집중 과목 불일치 설명 부족.

### P1. 놀라움 강화
- 기출 -> 개념 -> 기출 이력 구조를 SEW의 대표 경험으로 전면화.
- 개념 페이지 상단에 바로 풀기 CTA 추가.
- 진단 결과에서 "당신은 지금 무엇을 하면 되는가"를 한 문장으로 요약.

### P2. 재방문 루프 강화
- 홈 미션 완료 후 기록 화면에 성장 피드백을 더 명확히 표시.
- Daily 채점 후 "오늘의 다음 한 걸음"을 상단에 유지.
- 오답, 플래시카드, 개념, 기출을 하나의 복습 루프로 더 명시적으로 연결.

## 결론

SEW의 제품 가치는 이미 있다. 특히 기출과 개념의 연결은 임용생에게 실질적인 놀라움을 줄 수 있는 구조다. 현재의 가장 큰 병목은 기능 부족이 아니라 사용자 언어로 마감되지 않은 데이터 표현과 일부 숫자 신뢰성이다. 다음 개발은 새 기능 추가보다 "내부 식별자를 사용자 언어로 번역하고, 결과 화면의 다음 행동을 더 선명하게 만드는 작업"이 가장 효율적이다.
