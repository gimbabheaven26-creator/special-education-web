# 나다운 (nadaun) — 설계 문서

> IEP 기반 교수평일체화 보조도구. IEP 문서와 실제 수업 사이의 단절을 메운다.

## 1. 문제 정의

특수교사는 IEP를 작성하지만 그 문서는 행정 서류로만 남고, 실제 수업은 전혀 다른 형태로 진행된다. 다학년·다수준·다장애영역을 동시에 마주하는 특수교사에게 매 수업마다 학생 맞춤 수업을 준비하라는 것은 가혹하고, 실제 불가능하다.

교수평일체화(교육과정-수업-평가 일체화)가 교육의 핵심이지만, 특수교육 현장의 장벽이 이를 가로막는다:

- 학생이 계획대로 따라주지 않음
- 특수교육 교과서 품질 문제
- 교수학습 자료 제작 시간 부족
- 교사 역량 편차
- 적절한 수준의 무료 진단평가 도구 부재

## 2. 해결 방향

**이 도구의 존재 이유**: 매번 0에서 시작하지 않게 하는 것.

NISE가 공식 IEP 전산 시스템이므로, 나다운은 이를 대체하지 않는다. 나다운은 IEP 개념을 확장하여 **학생 맞춤형 체계화된 계획**을 세우는 보조 장치다. 계획이 올바르게 서야 수업도, 평가도, 성장도 이루어진다.

**접근법**: A안(AI 계획 생성기)으로 출발 → 검증 후 C안(하이브리드)으로 진화.

## 3. 사용자

- **핵심 사용자**: 일반학교 특수학급 특수교사
- **사용 시점**: 학기 시작 전(2~3월) 집중, 학기 중 수시 수정
- **학생 규모**: 교사당 6~7명/학기
- **교육과정**: 기본교육과정 (공통교육과정은 확장 영역)
- **교과**: 국어, 수학, 영어, 진로와직업 등
- **학부모 참여**: 미래 확장 (MVP 아님)

## 4-1. 개인정보 보호

학생 데이터(이름, 장애유형, 현행수준)는 민감한 교육 개인정보다.

### AI 전송 원칙: 교육적 정보만, 식별정보 제거

| AI에 보내는 것 | AI에 보내지 않는 것 |
|---------------|-------------------|
| 현행수준 서술 | 학생 이름 |
| 장애유형 (교육적 분류) | 생년월일 |
| 학년군 | 학교/반 |
| 교과/성취기준 | 보호자 정보 |
| | 주소/연락처 |

Claude API 호출 시 학생 식별정보를 프롬프트에 포함하지 않는다. 내부적으로 `student_id`로만 참조하고, AI에는 교육 계획 생성에 필요한 정보만 전달한다.

### 데이터 보호 정책

- **저장**: Supabase 기본 암호화 (at rest: AES-256, in transit: TLS)
- **접근 제어**: RLS로 담당 교사만 접근 (Section 6-1 참조)
- **보존 기간**: 학기 종료 후 1년 보관, 이후 교사 선택(보관/삭제)
- **AI 학습 제외**: Anthropic API 이용약관상 API 입력은 모델 학습에 사용되지 않음 (확인 필요)

## 5. 핵심 흐름 (MVP)

```
[1. 키움이 등록]
  이름, 학년, 장애유형, 현행수준(자유서술)
        ↓
[2. 교과 선택]
  기본교육과정: 국어/수학/영어/진로와직업 등
  학년군 선택 (초1~3, 초4~6, 중, 고)
        ↓
[3. 나다운 채비 (AI 계획 생성)]
  현행수준 + 성취기준 DB → Claude API가 초안 생성
  - 연간 목표
  - 단기 목표 (3~4개)
  - 주차별 계획 (단원/학습내용/성취기준 매핑)
  - 교수학습방법
  - 평가계획
        ↓
[4. 채비 다듬기 (교사 수정)]
  생성된 초안을 웹에서 직접 편집
        ↓
[5. 채비 내보내기]
  텍스트 복사 (NISE 붙여넣기용) / Excel / PDF
        ↓
[6. 저장]
  다음에 열어서 수정 가능, 학기 중 업데이트
```

## 7. AI 계획 생성 전략

### 프롬프트 구조

```
[시스템]
특수교육 기본교육과정 전문가. 교수평일체화 원칙에 따라 IEP 교과 계획 작성.

[입력]
- 교과, 학년군
- 현행수준 (교사 자유서술)
- 학기 주차 수
- 성취기준 목록 (DB에서 조회하여 주입)

[출력]
1. 연간 목표
2. 단기 목표 (3~4개)
3. 주차별 계획 (JSON)
4. 교수학습방법
5. 평가계획
```

### 핵심 원칙

- **AI는 창작자가 아니라 조합자** — 실제 성취기준 DB를 기반으로 배치하고 문장화
- 성취기준 DB가 선행 필수 (카이란이 소스 제공 → X가 수집/정제)
- DB 없이 AI만으로는 날조 위험 — 성취기준 코드를 만들어낼 수 있음

### API 호출 사양

| 항목 | 값 |
|------|------|
| 환경변수 | `ANTHROPIC_API_KEY` (서버 전용, 클라이언트 노출 금지) |
| 모델 | `claude-sonnet-4-20250514` (비용 효율 + 품질 균형) |
| Rate limit | 교사당 1일 30회 생성 (MVP) |
| 응답 방식 | 스트리밍 (SSE) — 생성 과정을 실시간으로 보여줌 |
| 타임아웃 | 60초 |
| 에러 처리 | 429 → "잠시 후 다시 시도" / 500 → "AI 서비스 점검 중" / timeout → 재시도 버튼 |
| 토큰 예산 | 입력 ~3,000 (성취기준 + 현행수준) + 출력 ~4,000 (계획 전체) ≈ 7,000/회 |
| 월 비용 추정 | 교사 50명 × 28회(7학생×4교과) × $0.02 ≈ $28/월 |

### 품질 로드맵

| 단계 | 방법 |
|------|------|
| 0차 (선행) | 기본교육과정 성취기준 DB 최소 구축 (국어/수학 중학교부터) |
| 1차 | 성취기준 DB + 현행수준 → Claude가 매핑하여 계획 생성 |
| 2차 | 교과서 단원 DB 추가 → 더 정확한 주차 배치 |
| 3차 | 교사 수정 데이터 축적 → 품질 향상 |

## 9. 기술 스택

| 항목 | 선택 | 이유 |
|------|------|------|

## 14. 확장 로드맵

```
MVP (v1): 나다운 채비
  AI 계획 생성 + 편집 + 내보내기
      ↓
v2: 나다운 자람
  교수학습 활동 제안 + 교과서 단원 연동
      ↓
v3: 형성평가 + 진단
  학생 성장 추적 + 무료 진단도구
      ↓
v4: 교수평일체화 완성
  IEP → 수업 → 평가 전 사이클 관통
```
